# Phase 9 — Continuous Validation / CI

**Branch:** `seo/09-ci`
**Goal:** Prevent regressions automatically. Every PR after this phase passes through SEO gates: Lighthouse CI, schema validation, broken-link check. Daily crons keep sitemaps fresh and pages indexable.

## Prerequisites

- Phase 8 merged.

## Steps

### 9.1 — GitHub Actions: Lighthouse CI per PR

`.github/workflows/lighthouse-ci.yml`:

```yaml
name: Lighthouse CI
on:
  pull_request:
    paths:
      - 'src/**'
      - 'app/**'
      - 'public/**'
      - 'next.config.js'
      - 'package.json'

jobs:
  lighthouse:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: pnpm install --frozen-lockfile
      - name: Wait for Vercel preview
        id: vercel
        uses: zentered/vercel-preview-url@v1.4.0
        with: { vercel_team_id: ${{ secrets.VERCEL_TEAM_ID }} }
      - name: Run Lighthouse CI
        uses: treosh/lighthouse-ci-action@v11
        with:
          urls: |
            ${{ steps.vercel.outputs.preview_url }}/
            ${{ steps.vercel.outputs.preview_url }}/suppliers/materialise-onsite
            ${{ steps.vercel.outputs.preview_url }}/suppliers
            ${{ steps.vercel.outputs.preview_url }}/best/sls-printing
            ${{ steps.vercel.outputs.preview_url }}/copenhagen/3d-printing
          configPath: .github/lighthouserc.json
          uploadArtifacts: true
          temporaryPublicStorage: true
```

`.github/lighthouserc.json`:

```json
{
  "ci": {
    "assert": {
      "assertions": {
        "categories:performance": ["error", { "minScore": 0.85 }],
        "categories:seo": ["error", { "minScore": 0.95 }],
        "categories:accessibility": ["warn", { "minScore": 0.9 }],
        "largest-contentful-paint": ["error", { "maxNumericValue": 2500 }],
        "cumulative-layout-shift": ["error", { "maxNumericValue": 0.1 }],
        "interaction-to-next-paint": ["error", { "maxNumericValue": 200 }]
      }
    }
  }
}
```

PR fails to merge if Lighthouse SEO < 95, performance < 85, or any CWV exceeds threshold.

### 9.2 — GitHub Actions: schema validation

`.github/workflows/schema-validate.yml`:

```yaml
name: Schema Validation
on:
  pull_request:
    paths: ['src/**', 'app/**']

jobs:
  schema:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: pnpm install --frozen-lockfile
      - name: Wait for Vercel preview
        id: vercel
        uses: zentered/vercel-preview-url@v1.4.0
      - name: Validate schema
        run: |
          pnpm tsx scripts/validate-schema-ci.ts \
            --base-url ${{ steps.vercel.outputs.preview_url }} \
            --urls /suppliers/materialise-onsite /suppliers /best/sls-printing /
```

`scripts/validate-schema-ci.ts`:
- Fetch each URL
- Extract all `<script type="application/ld+json">` blocks
- Validate each against Schema.org Validator API
- Fail with non-zero exit if any error

### 9.3 — GitHub Actions: broken internal links

`.github/workflows/broken-links.yml`: runs `lychee` over the preview URL, internal links only:

```yaml
- uses: lycheeverse/lychee-action@v2
  with:
    args: '--base ${{ steps.vercel.outputs.preview_url }} --no-progress --include-fragments --internal-only ${{ steps.vercel.outputs.preview_url }}'
```

Fails on any 4xx/5xx internal link.

### 9.4 — Pre-commit hook (Husky)

Install husky if not already: `pnpm dlx husky-init && pnpm install`.

`.husky/pre-commit`:

```bash
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

pnpm typecheck
pnpm lint --max-warnings=0

# Schema validation on staged tsx files that touch metadata or schema
STAGED=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(tsx|ts)$' || true)
if [ -n "$STAGED" ]; then
  echo "$STAGED" | grep -q "schema\|metadata\|generateMetadata" && {
    echo "Running schema-aware sanity checks..."
    pnpm tsx scripts/sanity-check-schema-imports.ts
  }
fi
```

`pnpm lint --max-warnings=0` enforces ESLint cleanliness on every commit.

### 9.5 — Daily cron: sitemap freshness + IndexNow ping

`vercel.json` cron entry, daily 02:00 UTC:

```json
{
  "crons": [
    { "path": "/api/cron/sitemap-refresh", "schedule": "0 2 * * *" },
    { "path": "/api/cron/indexnow", "schedule": "0 * * * *" },
    { "path": "/api/cron/page-health", "schedule": "0 3 * * *" }
  ]
}
```

`/api/cron/sitemap-refresh`: regenerates the sitemap-index children (or no-ops if Next.js sitemap routes are dynamic with revalidate). Mainly here as a safety net.

`/api/cron/page-health`: hits every `/suppliers/<slug>`, `/categories/...`, `/best/...`, `/<city>/<service>`, `/compare/...`, `/glossary/...` URL and asserts:
- HTTP 200
- `<script type="application/ld+json">` present
- `<title>`, `<meta name="description">`, canonical present
- No 404 / soft-404 indicators

Logs results to `page_health_runs` table. Alerts on regression (any URL that was 200 yesterday but is not 200 today).

### 9.6 — Update `seo/STATE.md` with rolling KPIs

A nightly Supabase function (`seo-state-updater`) writes a fresh `STATE.md` section appended at the bottom:

```markdown
## Rolling 30-day KPIs (auto-updated nightly)

- Indexed pages: 1,847 (+34 vs 30d ago)
- GSC avg position: 18.2 (was 24.3 at baseline)
- AI citation rate: 11% (was 4% at baseline)
- CWV pass rate (mobile): 92% (was 71% at baseline)

- Open broken-link issues: 0
```

This means every future `/seo` invocation reads fresh KPIs first.

### 9.7 — Open the PR

```bash
git add -A
git commit -m "seo(09): CI gates + daily crons for continuous validation"
git push -u origin seo/09-ci

gh pr create --title "seo(09): continuous validation / CI" --body "$(cat seo/PR_BODY.md)"
```

PR body checklist:
- [ ] Lighthouse CI workflow runs on this PR (and passes)
- [ ] Schema validation workflow runs on this PR (and passes)
- [ ] Broken-link workflow runs on this PR (and passes)
- [ ] Pre-commit hook fires locally on test commit
- [ ] Vercel cron entries deploy and `page-health` runs successfully
- [ ] `seo/STATE.md` populates with rolling KPIs after first nightly run

Update `STATE.md`, append phase summary to `seo/AUDIT.md`. **Final phase — print "SEO program complete; rolling KPIs now live in `seo/STATE.md`."**

## After Phase 9

Future `/seo` invocations:
- Default behavior: print rolling KPIs from `STATE.md`, summarize since last invocation, suggest next action.
- `/seo audit` re-runs the Phase 1 logic and produces a fresh `seo/AUDIT.md` (appended) with deltas.
- Single-page work (e.g. "fix SEO on /best/sla-printing") goes through normal PR flow without skill orchestration — the CI gates catch regressions.

## Anti-patterns (Phase 9 specifically)

- **Disabling failing CI checks to ship.** If Lighthouse fails on a PR, fix the regression — don't bypass.
- **Skipping the pre-commit hook with `--no-verify`.** Forbidden by CLAUDE.md global rules.
- **Crons without observability.** Every cron logs to a table. If cron silently stops firing, someone needs to notice — Phase 8's weekly digest checks cron freshness.

## Failure modes

- **Vercel preview not ready when CI runs** → bump the wait-for-preview action's timeout to 5 min.
- **Schema Validator API rate-limit hit by every PR** → use an internal validator (we have `schema-dts` types; build a local validator for common cases, fall back to API only for novel schemas).
- **Cron rate-limited by Supabase** → enable Supabase paid tier if free tier limits hit; sustainable cost ~$25/month for the SEO crons combined.
