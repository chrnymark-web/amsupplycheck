# Phase 5 — Programmatic SEO Expansion

**Branch:** `seo/05-programmatic`
**Goal:** Capture long-tail intent with new programmatic page types — each with unique data + intent, not template + keyword swap. Expand from ~430 URLs to several thousand high-quality, indexable pages without triggering thin-content / soft-404 penalties.

## Prerequisites

- Phase 4 merged (so generated content is approved and live for the seed suppliers).

## Steps

### 5.1 — New routes

Add four programmatic page families:

| Route | URL example | Generator gate |
|---|---|---|
| `app/[city]/[service]/page.tsx` | `/copenhagen/sls-printing` | ≥ 3 verified suppliers in city × service |
| `app/best/[service]/page.tsx` | `/best/sls-printing` | ≥ 5 verified suppliers in service globally |
| `app/compare/[pair]/page.tsx` | `/compare/xometry-vs-protolabs` | Pair listed in curated `seo/COMPARE_PAIRS.md` |
| `app/glossary/[term]/page.tsx` | `/glossary/laser-sintering` | Term listed in curated `seo/GLOSSARY.md` |

Strict gates are non-negotiable — without them, programmatic pages become thin/empty → soft-404 penalty.

### 5.2 — `/<city>/<service>` pages

Migration: `categories_seo` table augmented with `city_service_pages` view.

```ts
// app/[city]/[service]/page.tsx
export async function generateStaticParams() {
  const supabase = supabaseServer();
  const { data } = await supabase.rpc("city_service_combos_with_min_suppliers", { min_count: 3 });
  return (data ?? []).map((row) => ({ city: row.city_slug, service: row.service_slug }));
}
```

The RPC `city_service_combos_with_min_suppliers` returns rows like `{ city_slug, service_slug, supplier_count }` for combos meeting the threshold.

Page content:

- **H1:** "{Service} in {City} — {N} 3D printing service providers"
- **Intro (200+ words, generated via Phase 4 pipeline, reviewed via admin UI):** local context — what's the 3D printing scene in this city, who are the key players, what industries dominate locally.
- **Provider list:** ItemList JSON-LD + visible supplier cards, ordered by partnership tier then rating (existing ranking logic from `is_partner` cascade).
- **Internal links:** to parent service category, to nearby cities, to other services in this city.
- **Self-canonical:** never canonicalize to the parent category — these are distinct landing pages.

### 5.3 — `/best/<service>` pages

Lists "Best {service} 3D printing services" globally. Same gate (≥ 5 suppliers).

- **Methodology section** is mandatory: explicit, published criteria for ranking. E-E-A-T signal.
- **Comparison table**: sortable columns (location, materials, certifications, price range, lead time).
- **Per-supplier mini-card** with link to full profile.
- **`Article` JSON-LD** + `ItemList` of suppliers.
- **`author`** = SupplyCheck editorial team (named Person if a human curated the methodology).

### 5.4 — `/compare/<a>-vs-<b>` pages

Each pair listed in `seo/COMPARE_PAIRS.md` (curated, low-volume — start with 10-20 high-volume pairs from GSC top queries). Examples: xometry-vs-protolabs, materialise-vs-shapeways, hubs-vs-craftcloud.

- **Side-by-side comparison table** of every spec we have on both sides.
- **Pros/cons** generated by Phase 4 pipeline (reviewed before publishing).
- **Decision framework** ("Choose X if you need Y") — generated, reviewed.
- **Internal links** to both suppliers, to alternatives.
- **`@type: "ComparisonPage"`** does not exist in schema.org — use `Article` with descriptive headline.

### 5.5 — `/glossary/<term>` pages

50-100 3D printing terms. Each term lives in `seo/GLOSSARY.md` as `<slug>: <definition seed>`. Per term:

- **Definition** (50-150 words, generated, reviewed)
- **Related terms** (linked to other glossary entries)
- **Suppliers offering this** (if applicable — e.g. for `laser-sintering`, list SLS providers)
- **`DefinedTerm` JSON-LD** with `inDefinedTermSet` pointing to a glossary index page

### 5.6 — Sitemap-index split

Replace the single `app/sitemap.ts` from Phase 2 with a sitemap-index + per-type sitemaps:

```
app/sitemap.ts                         # sitemap-index pointing to children
app/sitemap/static/sitemap.ts          # /, /about, /browse, /search, /knowledge
app/sitemap/suppliers/sitemap.ts       # /suppliers/<slug>
app/sitemap/categories/sitemap.ts      # /categories/<type>/<slug>
app/sitemap/cities/sitemap.ts          # /<city>/<service>
app/sitemap/best/sitemap.ts            # /best/<service>
app/sitemap/compare/sitemap.ts         # /compare/<pair>
app/sitemap/guides/sitemap.ts          # /guides/<slug>
app/sitemap/glossary/sitemap.ts        # /glossary/<term>
```

Each child sitemap reads from Supabase. Update `lastmod` per row (`updated_at` or `seo_generated_at`).

If any single child exceeds 50,000 URLs, split it (`suppliers-1.xml`, `suppliers-2.xml`, etc.). For SupplyCheck's foreseeable scale, only `cities` could reach this.

Update `robots.txt`: `Sitemap: https://amsupplycheck.com/sitemap.xml` — single entry pointing at the index.

### 5.7 — Faceted nav handling

The supplier list, browse, and category pages have facets (sort, filter by tech, filter by material, filter by location, price range). Without controls, every facet combo is a separate URL → index bloat → thin content.

Rules:

- **Indexable facets** (canonical URLs, in sitemap, full content):
  - `/suppliers` (no facets)
  - `/categories/technology/<slug>` (single technology filter)
  - `/categories/material/<slug>` (single material filter)
  - `/categories/location/<slug>` (single location filter)
  - `/<city>/<service>` (city × service combo, gated)

- **Non-indexable facets** (`<meta robots="noindex,follow">`, NOT in sitemap, NOT canonicalized to a different page — they self-canonical):
  - `/suppliers?sort=rating`
  - `/suppliers?priceRange=high`
  - Any combination of 2+ filters via query string (`/suppliers?tech=sls&material=pa12`)

- **`rel="nofollow"`** on facet UI links (sort dropdowns, multi-select filter checkboxes) — Googlebot doesn't waste crawl budget on them.

Implementation: middleware reads query params, sets `noindex` header for non-indexable facet URLs. Facet UI uses `<a rel="nofollow">` on clickable filter chips.

### 5.8 — Content generation for new programmatic pages

Re-run the Phase 4 pipeline (`scripts/generate-seo-content.ts`) targeting:

- City × service combos in `city_service_combos_with_min_suppliers`
- "Best of" service pages
- Compare pairs from `seo/COMPARE_PAIRS.md`
- Glossary terms from `seo/GLOSSARY.md`

Same `pending_review` flow. Same admin UI (extended with a "page type" filter).

### 5.9 — Internal link densification

Every supplier page links to:

- Its city × service page (e.g. `/copenhagen/sls-printing`)
- Its top 2-3 service "best of" pages (e.g. `/best/sls-printing`)
- 3-5 related suppliers (existing logic — already implemented in `SupplierDetail.tsx`)
- Its parent technology category and material category pages

Every category page links to its peers. Glossary terms cross-link liberally.

The graph should make every supplier reachable in ≤ 3 clicks from `/`.

## Anti-patterns (Phase 5 specifically)

- **Removing the gates.** The ≥3 / ≥5 thresholds prevent thin-content pages — non-negotiable.
- **`canonical` to parent.** Each programmatic page self-canonicals. Canonicalizing `/copenhagen/sls-printing` to `/categories/technology/sls` would collapse the URL into the parent and waste the page.
- **Generating compare pages outside `COMPARE_PAIRS.md`.** Compare pages must be curated; auto-generating every supplier-pair combo would create N² thin pages.
- **Listing the same supplier in 50 different places without context.** Each programmatic page must have a unique angle (city, ranking, comparison context, glossary scope).

## Failure modes

- **Sitemap-index reference fails** (Google can't fetch a child) → check `next.config.js` rewrites, ensure each child URL returns 200 + `Content-Type: application/xml`.
- **Programmatic page set explodes the build time** → reduce `generateStaticParams` cardinality (top N cities, top N services), set rest to `dynamic = "force-dynamic"` or `dynamicParams = true` with ISR.
- **Generated content fails review at scale** → reduce batch size, tighten prompt, add a final AI-tells linter step before saving.
